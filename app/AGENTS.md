# app — native multiplatform SwiftUI client

## Purpose

A native SwiftUI app for iPhone, iPad, and Mac: a thin-but-cached client over the FastAPI REST boundary (`../src/cookbook_kb/api/`). The server owns intelligence and data; the app owns presentation, local cache, and offline resilience. Chosen over a Streamlit/web UI so it can be a real multiplatform product with a SwiftData cache and async ingestion.

## Ownership

- `CookbookApp/` — the buildable app: XcodeGen project + the two platform targets (iOS `Cookbook`, macOS `CookbookMac`) sharing `Sources/`. Child DOX (build + Info.plist contracts).
- `CookbookKit/` — the SwiftPM package holding all reusable code, split into two products:
  - `Sources/CookbookKit/` — logic: networking, DTOs, SwiftData persistence, stores, sync. Child DOX.
  - `Sources/CookbookUI/` — views: screens, components, theme. Child DOX.
  - `Tests/CookbookKitTests/` — package tests.

## Local Contracts

- **SwiftData repository pattern is mandatory — NEVER `@Query` in views.** With background sync/batch writes, `@Query` merges changes to the main context synchronously and freezes the UI. The required pattern:
  1. Value-type `Sendable` DTOs mirror display data.
  2. An `@Observable @MainActor` store fetches off-main, maps to DTOs, publishes arrays.
  3. Views bind to the store via `@Environment` — never to `@Query` or `@Model`.
  4. `store.refresh()` is called explicitly after sync/mutation — never reactively.
  5. The background sync actor writes SwiftData freely; the UI sees changes only on refresh.
  `@Query` is acceptable ONLY if the sole write path is direct user interaction on the main context (no background sync). This app has sync — so: no `@Query`.
- **Image loading uses Nuke (`LazyImage`/NukeUI).** Never hand-roll async image loading — it floods the network and lacks caching/cancellation.
- **The app is a thin client.** Don't reimplement server logic (search, planning, nutrition) on-device; call the endpoint. Cache results in SwiftData for offline/perf.
- **Respect the rate-sensitive embed endpoint.** Client search behavior must not hammer `/recipes/semantic` (each is a `jina` embed). See `CookbookKit/Sources/CookbookUI/AGENTS.md` for the as-you-type debounce/min-length/dedupe rules.
- **Destructive + add flows are app-wide contracts.** Recipe/library/import-job deletes are always behind a `.confirmationDialog` and reconcile from the server response (catalog version), never a blind pull. The **Assistant is the unified ask + ADD surface**: describe / find-online / URL / PDF → an inline draft that **never persists until an explicit Save**. Specifics live in `CookbookKit/Sources/CookbookUI/AGENTS.md` (delete semantics + the compose draft/Save contract).

## Work Guidance

- Project is generated by XcodeGen from `CookbookApp/project.yml`; regenerate after target/plist changes.
- Source of truth is the Swift compiler / `xcodebuild` / `swift build` — **SourceKit "No such module" / "Cannot find type" diagnostics are persistent false positives (lag); ignore them and trust the build.**
- The iOS app trails the backend in features but is the reference for native patterns; check it before porting.
